## 真实性的提高

### 目标

在尽量不增加完整分析耗时的前提下提升拟真性：优先复用已经生成的消息、segment summary、topic、persona、snapshot，在推演/实时会话启动时做更聪明的检索、标注和上下文组装；只有收益明确时才增加可缓存、可异步、可关闭的分析产物。

### 1. 拟真性基线与问题归因

- [ ] 建立拟真性失败样例集：挑选真实聊天里的关键改写点，记录当前推演常见失真类型，如过度乐观、忽视未来客观事实、对方突然变得太会表达、上下文检索不相关、短链轮次不连贯。
- [ ] 定义可验证指标：未来信息泄漏为 0；输出不得明显违反 persona；回复长度/语气/推进强度应与真实对方习惯一致；当未来客观事实显示高概率失败时，分支不能轻易翻盘。
- [ ] 补充回归测试或离线评估脚本：用固定上下文包验证 prompt 与检索策略不会把"未来发生的事实"当作角色已知事实。

### 2. 客观事实认知提升：全量知识分层检索

- [ ] 将推演上下文明确分成三类：**已发生事实**（cutoff 前，角色可知）、**未来客观事实**（cutoff 后，只有模型用于判断稳定约束，角色不可知）、**分支内新事实**（用户改写后实时生成）。
- [ ] 在 context pack 中加入 `future_evidence_digests`：从 cutoff 后的 topic、segment summary、snapshot 中检索能约束人格/关系走向的证据，并显式标记为"建模证据，不可被角色引用"。
- [ ] 修改 branch assessment / first reply / next turn prompt：允许未来客观事实影响概率、风险和保守程度，但禁止生成角色不可能知道的信息。
- [ ] 优先使用现有 topic/snapshot/summary，不新增完整分析 LLM 阶段；检索在推演时完成，控制 token budget 和条目上限。
- [ ] 增加泄漏测试：当未来事实包含"对方明确拒绝/已有偏好/关系已降温"时，LLM 可以据此降低成功概率，但回复文本不得直接说出 cutoff 后才出现的信息。

### 3. 人格提取增强：更完整但不拖慢主分析

- [ ] 保留现有全量 persona 作为人物稳定画像，同时补充"截止前 persona slice"与"未来佐证片段"的区别，避免所有时间线混成角色当下认知。
- [ ] 用确定性统计补充表达风格：平均消息长度、短句比例、连续发送习惯、常用语气词/标点、沉默/回应间隔区间；这类统计不调用 LLM，不增加分析瓶颈。
- [ ] 将 persona 输出拆成更可执行的约束：表达密度、冲突反应、推进边界、亲密表达上限、面对当前对象的特殊模式。
- [ ] 为 persona 增加证据定位：每条关键模式关联 segment/topic/snapshot id，推演时只取最相关的少量证据。

### 4. 知识检索策略升级

- [ ] 现阶段先用结构化检索：按 topic 相关度、时间距离、敏感状态、关系阶段相似度排序，不先引入 embedding 以免增加索引和依赖复杂度。
- [ ] 对推演上下文设置硬预算：当前段原文优先，其次 cutoff 前 snapshot/persona，再加入少量高相关 future evidence；避免把全量摘要塞进 prompt。
- [ ] 将"相关但过远"的旧证据降权，将"与目标改写同一敏感主题"的证据升权。
- [ ] 预留 embedding 检索为后续增强：只有当结构化 topic 检索召回不足时，再做异步索引或可选重建，不阻塞主分析链路。

### 5. 将短链推演升级为实时分支会话

- [ ] 新增分支会话模型：保存起点消息、改写内容、上下文包快照、当前分支状态、分支内消息 transcript。
- [ ] 后端提供追加用户消息接口：用户只扮演 self，LLM 只扮演 other；每个分支会话同一时间只允许一个 LLM 回复任务运行。
- [ ] 每轮回复都携带完整分支 transcript、当前 branch state、persona、相关 topic 和分层事实证据；LLM 只生成对方下一批消息，不替用户说话。
- [ ] 保留现有 `single_reply` / `short_thread` 作为兼容入口，实时会话作为新的主体验逐步替代短链推演。

### 6. 实时聊天拟真交互

- [ ] 用户输入窗口合并：用户连续发送多条消息时，等待约 5 秒无新消息再触发 LLM，避免对方抢答导致上下文错位。
- [ ] LLM 回复串行：上一轮 LLM 未完成时，新用户消息进入队列或等待下一窗口，禁止并行生成多个互相不知道的回复。
- [ ] 回复拆泡与延迟发送：LLM 输出按短句拆成多条气泡，并按长度、标点和表达密度模拟 1-3 秒发送延迟。
- [ ] 显示输入中状态：在等待窗口和生成阶段展示轻量 typing 状态，但不提前暴露未完成内容。

### 7. 综合难题与解决办法

- [ ] 难题：越用全量历史，越容易泄漏未来。解决：在数据结构和 prompt 中硬分层，只允许未来事实影响"概率/风险/保守程度"，不允许成为角色台词。
- [ ] 难题：更完整的人格通常意味着更多分析成本。解决：优先复用现有全量 persona，新增低成本确定性风格统计和按需检索，不把主分析链路再次变重。
- [ ] 难题：检索内容越多，推演越慢且越容易跑偏。解决：做排序、预算、证据类型配额和上下文快照，先保证少而准。
- [ ] 难题：短链自动续写天然不真实，因为用户没有参与下一轮。解决：把主体验改为实时分支会话，用户真实输入 self，LLM 只负责 other。
- [ ] 难题：实时会话如果并发，会破坏对话连贯性。解决：分支会话级串行锁、输入窗口合并、完整 transcript 续写。
