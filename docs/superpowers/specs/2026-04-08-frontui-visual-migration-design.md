# frontUI 视觉整迁到 desktop 的设计说明

日期：2026-04-08  
状态：已评审通过，待用户复核文档后进入 implementation plan

## 1. 背景

当前 `desktop` 子工程已经具备真实可运行的 Electron + React + TypeScript 桌面前端，并且已接入：

- Electron preload / IPC bridge
- 本地 Python API / worker 启动状态
- 设置保存
- 聊天记录导入
- 会话列表与消息读取
- 分析 / 推演 / 分支视图相关流程

但当前桌面端视觉仍然只是“产品化壳层”，并未与用户提供的 `D:\frontUI\src` 保持一致。用户新的明确要求是：

1. **功能逻辑继续沿用当前 desktop 项目这套。**
2. **视觉效果优先完全向 `frontUI` 看齐。**
3. **优先采用 frontUI 现成页面骨架，而不是仅做风格临摹。**
4. **暂时接不上的能力允许先用 mock 顶住。**
5. **哪些已经真实接入、哪些仍是 mock，不在前端展示，只在文档中维护清单。**

## 2. 目标

本阶段目标不是“继续打磨当前 desktop 壳层”，而是：

> **把 `frontUI` 作为新的桌面主界面视觉母版，优先保证主聊天界面在观感和结构上尽量与 `frontUI` 一致，再逐步把已有真实业务能力一点点接回去。**

成功标准：

- 主聊天界面整体布局、层级、色彩、聊天列表、聊天窗口、输入区与 `frontUI` 高度一致。
- 现有 `desktop` 的真实业务能力不被放弃，只是暂时允许部分入口先用 mock 过渡。
- 项目中有一份清晰文档记录“真实接入 / mock 占位 / 待接入”状态。

## 3. 非目标

本阶段暂不追求以下事项一次性完成：

- 所有外围弹层与分析页面立刻完成 frontUI 化
- 一次性清除所有 mock
- 一次性重写全部测试
- 保证目录结构与 `frontUI` 完全逐文件一致

重点是：**优先拿下主界面视觉的一致性。**

## 4. 采用方案

采用“**方案 1：直接整体搬运 frontUI 的页面骨架**”。

核心思路：

1. 直接把 `frontUI` 的核心页面结构引入 `desktop`：
   - `Sidebar`
   - `ChatList`
   - `ChatWindow`
   - 对应样式体系与组件拆分思路
2. 在 `desktop` 中建立一层适配层，把真实后端数据转换成 frontUI 组件需要的数据形状。
3. 对于暂时没有真实接法或前端骨架尚未嵌回的能力，用 mock 保证页面完整和交互不至于中断。
4. 用文档而不是前端标签维护实现状态清单。

## 5. 页面与能力分层

### 5.1 主聊天界面（本阶段重点）

主聊天界面将尽量按 `frontUI` 原样落地，包括：

- 左侧深色图标导航栏
- 中间聊天列表栏
- 右侧聊天窗口
- 顶部聊天标题区域
- 聊天气泡、时间分组、头像布局
- 底部输入区、工具栏、发送按钮

### 5.2 外围能力（本阶段允许过渡）

以下能力本阶段不强求与 `frontUI` 视觉完全一致，但必须保留可继续工作的路径：

- 欢迎引导
- 模型设置
- 导入聊天记录
- 启动状态页 / boot screen
- 分析侧栏
- 改写 / 推演面板
- 分支视图

这些能力在第一阶段可以采用以下策略之一：

- 继续复用现有实现
- 从主界面提供临时入口
- 在新主界面中用 mock 占住未来位置

## 6. 数据接入策略

为避免把 `frontUI` mock 逻辑直接写死在桌面产品里，数据将分成三层：

### 6.1 真实数据层

来源于当前已存在的 `desktop` 服务与 bridge：

- 启动状态：`desktop bridge` + local backend health
- 会话列表：后端 conversations API
- 消息列表：后端 messages API
- 设置：settings API
- 导入：desktop import bridge + imports API

### 6.2 视觉适配层

新增一层 frontUI 专用 view model 适配：

- `ConversationRead -> FrontChatListItem`
- `MessageRead -> FrontChatMessage`
- `SettingRead -> frontUI-compatible setting forms / statuses`

这一层负责把真实业务数据翻译成 `frontUI` 风格组件直接可消费的结构，避免业务层直接依赖视觉细节。

### 6.3 Mock 过渡层

对于未接回真实实现的区域，允许存在临时 mock，例如：

- `frontUI` 原有但当前产品未完成的视觉入口
- 新主界面中暂时还没有真实数据或真实行为映射的按钮
- 尚未嵌回的分析 / 推演面板位置

要求：

- mock 只作为过渡，不得伪装成最终真实能力
- 文档中必须记录 mock 区域
- 后续 implementation plan 需要明确替换顺序

## 7. 迁移范围

### 7.1 直接迁移内容

优先直接吸收以下 `frontUI` 资产：

- `src/app/App.tsx` 的主界面结构
- `src/app/components/Sidebar.tsx`
- `src/app/components/ChatList.tsx`
- `src/app/components/ChatWindow.tsx`
- `src/app/mockData.ts` 中体现的 view model 形状
- `src/styles/index.css` 及其关联样式体系思路

### 7.2 有选择迁移内容

`frontUI/src/app/components/ui/*` 数量较多，本阶段不做“全量引入”。

原则：

- 仅在当前主界面实际需要时才引入
- 优先引入轻量且可控的依赖
- 避免为了一个页面把整套无关 UI 库搬入 `desktop`

### 7.3 暂不迁移内容

如果 `frontUI` 中存在与当前产品目标无关、或需要大幅改造才可落地的内容，本阶段不迁移。

## 8. 依赖策略

用户已明确允许为高保真迁移补充前端依赖。

因此本阶段允许：

- 补充 `frontUI` 主界面实际依赖的前端库
- 补充必要样式与图标依赖
- 在不影响可维护性的前提下调整 `desktop/package.json`

但仍需遵守：

- 不引入明显无用的大型依赖集合
- 优先补最小依赖闭包
- 每个新增依赖都必须服务于当前迁移目标

## 9. UI 与业务的边界约束

为了避免视觉迁移反向污染业务逻辑，需遵守以下约束：

1. **frontUI 组件不能直接持有桌面端业务协议细节。**
   - 不直接耦合后端 API response shape
   - 不直接耦合 Electron bridge 细节
2. **业务状态机继续保留在 `desktop` 现有应用层。**
3. **frontUI 组件优先作为展示层 / 交互层使用。**
4. **真实功能回接通过 adapter / container 进行。**

## 10. 实现状态文档要求

需要新增或更新一份文档，记录以下内容：

- 已经真实接入的能力
- 当前仍由 mock 驱动的能力
- 暂无视觉入口但已有真实实现的能力
- 下一步替换 mock 为真实实现的顺序

文档要服务于后续查漏补缺，因此要求：

- 用表格维护
- 每个条目写清“前端入口 / 数据来源 / 当前状态 / 下一步”
- 不在前端 UI 中展示这些标记

## 11. 验证要求

本阶段完成后，至少要验证：

1. `desktop` 仍然可以正常开发启动
2. Electron 主进程 / preload 产物不受影响
3. 前端测试与类型检查可恢复到通过状态
4. 主界面视觉明显切换到 `frontUI` 风格
5. 至少保证“启动 -> 进入主界面 -> 查看会话 / 消息”链路可运行

必要验证命令包括：

- `cd desktop && npm test`
- `cd desktop && npm run typecheck`
- `cd desktop && npm run build`
- 本机手动运行 Electron 桌面壳进行 GUI 验证

## 12. 风险

### 风险 1：视觉迁移太快，真实能力入口丢失

缓解：先记录现有所有功能入口，迁移时逐项核对，不允许静默丢失。

### 风险 2：frontUI 原本基于 mock，迁入后与真实数据形状不匹配

缓解：通过 adapter 层隔离，不让展示组件直接吃原始 API 数据。

### 风险 3：为迁移引入过多依赖，导致桌面工程变重

缓解：只引入主聊天界面实际用到的依赖，不做整包照搬。

### 风险 4：主界面完成后，外围弹层视觉不一致

缓解：接受这是阶段性结果，并在后续计划中列为第二阶段任务，而不是在第一阶段混做。

## 13. 分阶段落地建议

### 阶段 1：主界面骨架替换

- 引入 frontUI 结构与主样式
- 让 desktop 主界面在视觉上变成 frontUI 风格
- 暂时允许使用 mock 数据保障页面完整性

### 阶段 2：真实会话 / 消息回接

- 把会话列表和消息窗口接回真实 API
- 保证主要浏览链路跑通

### 阶段 3：功能入口回植

- 把设置、导入、欢迎页等入口安置到新壳中
- 校正交互路径

### 阶段 4：高级能力嵌回

- 分析侧栏
- 改写 / 推演
- 分支视图

## 14. 最终决策

本次视觉迁移的执行原则如下：

- **视觉优先以 frontUI 为准，不再以当前 desktop 壳层为准。**
- **功能逻辑继续以当前 desktop 为准。**
- **短期允许 mock 过渡，但必须有文档可追踪。**
- **第一阶段重点是“看起来先对”，然后再把真实能力一项项接回去。**
