# V2 低工作量界面修整设计

日期：2026-04-12

## 1. 背景

`plan/TODO.md` 已经把上一轮桌面端细节修整归档为 `V1`，并新增了一组 `V2` 迭代任务。

本次先处理其中 5 个低工作量问题：

1. 消息气泡圆角修复
2. 改写的消息气泡颜色变化
3. 聊天记录的时间展示
4. 导入界面的“导入后立即开始分析”选项
5. 导入后的消息记录展示

这些需求都属于现有桌面前端交互和视觉层的小范围修补，不需要改动后端数据模型，也不应借机扩展成大功能开发。

---

## 2. 目标

本轮完成后必须满足：

- frontUI 消息气泡不再出现一个尖角、三个圆角的主体形态
- 被用户改写的那条目标消息拥有独立且醒目的材质表现
- 聊天时间展示符合用户给出的微信式规则
- 导入弹窗把“立即分析”从 checkbox 改成下拉选项
- 新导入会话首次进入时，消息区默认停在最近消息位置

---

## 3. 非目标

本次不包含：

- 聊天记录弹窗、搜索、日期筛选等高工作量功能
- 导入接口协议或后端导入逻辑变更
- 对现有 frontUI 架构做无关抽象或重构
- 对消息列表分页策略做全面重写

---

## 4. 方案比较

### 方案 A：定点修补 + 共享时间格式化工具

做法：

- 在现有 frontUI 渲染和 view-model 层做小范围修改
- 仅为“时间展示”引入一个共享格式化函数
- 其余需求沿现有状态流定点修补

优点：

- 风险最低
- 与“先清理一批小 bug”的目标一致
- 不会把简单修复扩展成结构性重构

缺点：

- 仍然保留现有组件边界，不会顺手统一更多视觉逻辑

结论：

- 采用该方案

### 方案 B：顺手整理消息样式与导入状态层

做法：

- 把气泡视觉、消息时间、导入选项、滚动策略都抽成更多复用层

优点：

- 后续演进理论上更统一

缺点：

- diff 会明显扩大
- 容易引入与本轮需求无关的回归

结论：

- 不采用

### 方案 C：提前为聊天记录弹窗预埋能力

做法：

- 在低工作量任务中顺手把搜索、定位、日期筛选的数据能力一并加上

优点：

- 看似能为后续高工作量任务铺路

缺点：

- 当前需求边界会被拉宽
- 容易把一轮小修做成半个新功能

结论：

- 不采用

## 6. 详细设计

### 6.1 消息气泡圆角修复

当前 `desktop/src/frontui/ChatWindow.tsx` 中的消息主体通过：

- `rounded-lg rounded-tr-none`
- `rounded-lg rounded-tl-none`

来制造“微信气泡尾部”观感，但结果是主体本身出现一个尖角。

本次调整原则：

- 气泡主体改为四角圆角
- 不移除现有箭头/尾部装饰概念
- 尾部装饰不能重新破坏主体四角圆角

实现策略：

- 将主体 class 收口为统一的四角圆角形态
- 箭头继续作为独立装饰定位，而不是依赖主体裁掉某一个圆角
- 同步修正相关测试断言，避免再检查 `rounded-tl-none` / `rounded-tr-none`

### 6.2 改写目标消息的特殊材质

当前改写链路里：

- 原始目标消息文本会在 `App.tsx` 中被替换成用户改写后的文本
- 但该消息没有明确的独立视觉语义
- 推演结果的粉/蓝气泡只代表“这是推演产物”，并不代表“这是改写起点”

本次设计要求：

- 被改写的目标消息必须拥有单独视觉态
- 该视觉态只作用于那一条被改写的历史消息
- 该视觉态与推演结果的 `simulation-self / simulation-other` 区分开

实现策略：

- 为 frontUI 消息模型新增独立 `bubbleTone` 值，例如 `rewrite-target`
- 由 `App.tsx` 在构造当前会话消息时，给目标消息打上独立标记
- `ChatWindow.tsx` 根据该标记渲染特殊材质

视觉方向：

- 仍保持消息气泡原本形状
- 使用青蓝到银白的轻微渐变底色，叠加柔和外发光与细微流动高光，表达“分支起点”
- 不改变对话阅读方向，不把它做成卡片或横幅

### 6.3 微信式时间展示

当前 `desktop/src/lib/frontUiAdapters.ts` 的 `resolveTimestampLabel()` 只提取 `HH:mm`，因此所有消息都缺少日期语义。

本次统一规则：

- 当天消息：`13:17`
- 昨天消息：`昨天 13:17`
- 前天消息：`前天 13:17`
- 本年其他消息：`3月1日 13:17`
- 往年消息：`2025年3月1日 13:17`

实现策略：

- 将现有简化版 `resolveTimestampLabel()` 升级为真正的聊天时间格式化工具
- 该工具只负责给聊天 UI 输出 label，不改变底层时间原值
- 真实消息、推演消息、本地临时消息统一复用同一套格式化逻辑

设计约束：

- 使用本地时区解释消息时间
- 保持纯函数形式，便于在测试中传入固定“当前时间”
- 不影响现有 `timestampRaw`

### 6.4 导入界面的“立即分析”选项

当前 `ImportDialog` 用 checkbox 渲染：

- 一个勾选框
- 一段说明文案

用户要求改为单个下拉框选项：

- `只导入`
- `导入并分析`

默认值：

- `只导入`

实现策略：

- UI 层使用枚举型下拉框表示导入模式
- 提交时再转换为现有 `autoAnalyze: boolean`
- 因为后端接口已经支持 `auto_analyze` 布尔值，所以不改 API 协议

文案策略：

- 下拉框标签直接表达动作
- 下方说明文案根据当前选项切换

### 6.5 导入后默认滚到底部

当前消息加载流程在 `App.tsx` 中通过：

- 首次读取最新一页消息时使用 `order=desc`
- 前端反转后按时间正序渲染

`ChatWindow.tsx` 也已经在以下情况下尝试滚动到底部：

- conversation 变化
- 追加新消息
- 空列表首次变成有消息

但在“刚导入一个新会话并首次渲染消息”这一场景下，现有判定仍可能停在中段。

本次修复原则：

- 新导入会话首次拿到最新消息批次后，默认滚动到底部
- 用户主动上滑加载更早消息时，仍保持原有锚点恢复行为
- 不通过粗暴的每次 render 都滚到底部来掩盖问题

实现策略：

- 保持“向上加载更早消息”与“首次进入最新消息视图”两类滚动意图分离
- 将“首次最新页成功渲染”显式视为应滚到底部的场景
- 仅在 prepend 旧消息时使用现有锚点恢复逻辑

## 8. 风险与约束

主要风险：

- 时间标签一旦变更，会影响“相邻消息是否合并显示时间”的现有判定
- 改写目标消息若直接复用现有推演气泡色，会造成视觉语义混淆
- 滚动修复若处理不当，可能破坏“加载更早消息”后的阅读位置稳定性

应对策略：

- 时间格式化先通过纯函数测试锁定规则
- 改写目标消息采用独立视觉态，不复用 simulation 色板
- 滚动逻辑只修正首次最新页定位，不改 prepend 旧消息的锚点恢复机制

---

## 9. 实施结论

本轮 V2 低工作量任务应当按“前端定点修补”执行，而不是扩展成架构性整理。

正确方向是：

> 在不改后端协议和不扩大范围的前提下，修正消息气泡、改写目标视觉、时间标签、导入弹窗交互以及导入后的首屏滚动行为，并用前端测试锁定这些桌面端细节。
