# 桌面前端视觉精修与 Figma 气质对齐设计

日期：2026-04-08

## 1. 目标

在已经完成 Task 5 的桌面前端基础上，对 `desktop/` 做一轮 **产品级视觉精修**，把当前“功能已通的桌面壳”提升到更像 Windows 本地聊天客户端的完成度。

本轮目标不是新增业务能力，而是统一和打磨以下区域的视觉与轻交互：

- 主界面整体（三栏布局）
- 欢迎弹窗
- 设置抽屉
- 导入弹窗
- 改写面板 / 分支视图 / 分析侧栏

本轮同时明确后续节奏：**视觉精修完成后，尽快修复后端接口/数据层中使用废弃方法导致测试出现 warning 的问题**。

## 2. 设计边界

### 2.1 本轮包含

- 对齐 `D:\frontUI\src` 的整体气质与骨架，但不直接迁移其原型代码
- 统一桌面窗口感、三栏关系、颜色体系、边框、圆角、阴影、输入控件、按钮风格
- 精修会话列表、聊天主区、分析增强层、欢迎弹窗、导入弹窗、设置抽屉
- 在不改变主流程的前提下，允许顺手优化轻量交互细节

### 2.2 本轮不包含

- 新增业务流程
- 重做状态管理或数据流
- 引入新的 UI 组件库 / CSS 体系 / 路由体系
- 直接复制 `D:\frontUI\src\app\components\ui\*`
- Windows 安装器、自动更新、正式 release 打包
- 后端 warning 修复本身（该项作为本轮之后的紧急跟进任务）

- 实施优先级：**D** —— 整体一致性优先
- 改动程度：**B** —— 允许轻量交互优化，但不改核心流程

## 4. 方案比较

### 方案 A：纯样式层精修

只调整 `styles.css` 和现有 class 呈现，不改变任何组件结构。

优点：
- 风险最低
- 上手最快

缺点：
- 很难真正做出“桌面窗口感”
- 弹窗 / 抽屉 / 分析侧栏 与主界面的统一性有限
- 会话列表与聊天主区仍容易保留较强的 Web 卡片感

结论：
- 不采用

### 方案 B：小幅结构整理 + 全局视觉统一

允许对少量组件结构、header 区、容器关系做轻量调整，同时统一视觉语言。

优点：
- 能显著提升完成度
- 与当前“聊天主导、分析增强”的产品目标最匹配
- 不会演变成前端重做

缺点：
- 改动量略高于纯样式修补
- 需要补少量回归验证

结论：
- **采用该方案**

### 方案 C：深度贴近 Figma 导出稿

尽量把 renderer 做到接近 `frontUI` 的结构和视觉。

优点：
- 视觉变化最大
- 最容易“看起来像另一版成品”

缺点：
- 易把原型语义带回产品代码
- 风险高，容易影响已完成的 Task 3/4/5 结构
- 容易扩大为一次前端重构

结论：
- 不采用

## 5. 核心设计结论

### 5.1 本轮不是“迁移 frontUI”，而是“用 frontUI 重塑 desktop renderer 的视觉语言”

`D:\frontUI\src` 的作用限定为：

- 提供三栏比例、深浅区域关系、聊天主区氛围、桌面 IM 气质
- 不提供业务边界
- 不提供真实状态管理方式
- 不提供可直接上线的组件实现

因此本轮应：

- 参考其骨架与色彩关系
- 丢弃其 `mockData` 与即时聊天器假设
- 用当前真实产品语义（导入 / 浏览 / 分析 / 改写 / 推演）重新组织界面

### 5.2 主视觉定位：桌面聊天客户端，而不是 SaaS 后台

本产品本轮应更像：

- Windows 本地聊天工具
- 历史聊天浏览器
- 带高级分析工具的桌面客户端

而不是：

- 数据后台
- 报表系统
- 通用 AI chat app

### 5.3 分析能力必须“清晰可见，但不喧宾夺主”

改写面板、分支视图、分析侧栏都要带专业工具感，但层级应低于聊天主区：

- 聊天主区始终是主舞台
- 分析信息是增强视角
- 分支结果是聊天的特殊态，而不是另一个独立页面

## 6. 整体布局设计

### 6.1 左栏：深色桌面导航骨架

目标：更像原生桌面应用的功能侧栏。

要求：
- 深灰黑底
- 图标为主
- 激活态用绿色强化
- 顶部保留产品身份区，不使用高噪音装饰
- 底部保留设置等系统级入口

视觉原则：
- 稳定、克制、低噪音
- 少做卡片和浮层感
- 像桌面程序外框骨架

### 6.2 中栏：聊天客户端式会话列表

目标：从“卡片列表”转为“桌面 IM 会话栏”。

要求：
- 顶部搜索框与操作按钮更收敛
- 会话项更扁平，减少卡片堆叠感
- 激活项以背景层级变化 + 局部强调表现
- 分析状态融入会话条目，而不是抢眼漂浮 badge
- 搜索、空态、hover、active 语义统一

信息优先级：
1. 联系人 / 会话标题
2. 最近消息摘要
3. 时间 / 分析状态

### 6.3 右栏：聊天主舞台

目标：成为整个桌面端最像成品的区域。

要求：
- 顶部标题区更像桌面聊天窗口 header
- 历史消息区更接近本地聊天记录浏览器
- 自己消息与对方消息形成明确但克制的区分
- hover 动作自然嵌入消息流，不像网页工具按钮
- 空态、错误态、加载态与整体风格一致

### 6.4 桌面窗口感

要求：
- 做出明显窗口内容层次，但不强做完整 Windows 标题栏仿真
- 收敛网页氛围背景，增强“本地程序容器感”
- 用更稳定的边线、分区、背景层次代替过强的网页投影和营销式渐变

---

## 7. 颜色与视觉语言

### 7.1 颜色角色

主色基调调整为：
- 主背景：灰白 / 浅灰
- 左栏：深灰黑
- 强调色：绿色
- 日常信息：中性灰
- 分析增强：低饱和辅助色，不再以蓝色主导

### 7.2 颜色使用规则

- 绿色：当前激活态、主操作、完成态、重要确认
- 灰：结构边界、次级信息、默认背景、未激活状态
- 白：内容承载区、消息主体、弹层主容器
- 深色：导航和整体骨架

### 7.3 风格目标

从当前偏蓝科技感切到：
- 更接近微信 PC / 本地桌面聊天工具
- 更少 dashboard 感
- 更少“网页产品宣传感”
- 更强“日常使用的本地聊天工具”感

---

## 8. 重点区域设计

### 8.1 WelcomeModal

目标：像桌面应用首次使用引导，而不是网页营销弹窗。

要求：
- 两个主动作清晰：配置模型、导入聊天记录
- 产品说明简短克制
- 容器、按钮、标题风格与主界面统一
- 支持关闭，但关闭后仍可从设置入口继续操作

### 8.2 ImportDialog

目标：像桌面工具的导入向导。

要求：
- 文件选择区更突出
- 已选文件信息更易读
- `self_display_name` 的说明更自然
- 提交动作更像“导入任务启动”而非普通表单提交

### 8.3 SettingsDrawer

目标：像桌面应用设置面板，而不是简单表单抽屉。

要求：
- 表单分组层次更明确
- 输入控件统一
- 主次按钮清晰
- 为将来本地模型入口预留视觉空间，但本轮不扩业务字段

### 8.4 RewritePanel

目标：像单条消息的“编辑与推演工作面板”。

要求：
- 原消息、发送时间、改写内容形成清晰对照
- 模式选择与轮数控制更规整
- 主动作“开始推演”强调清楚
- pending / disabled / error 状态与整站风格一致

### 8.5 BranchView

目标：让用户感觉仍在原聊天语境中浏览另一个可能分支。

要求：
- 顶部明确提供“返回原始历史”
- 强化原消息、改写内容、首轮回复、短链 turns 的阅读顺序
- 不把分支视图设计成新的独立页面风格

### 8.6 AnalysisInspector

目标：像专业辅助侧板，而不是第二主舞台。

要求：
- `topics / persona / snapshot` tab 清晰但克制
- 宽度与层级收敛，不挤压聊天主区
- 信息展示结构化，但不做成管理后台表格感
- loading / empty / error 语义稳定

## 9. 轻交互调整边界

本轮允许的轻交互优化：
- hover 显示的消息动作更自然
- 列表项 hover / active 更明确
- 弹窗与抽屉的按钮主次关系更清楚
- 分析面板 tab 切换视觉更稳定

本轮不允许的交互扩大：
- 不改导入流程语义
- 不改会话加载与分析主流程
- 不改 branch / inspector 的核心业务逻辑
- 不增加新的导航路由或多页面结构

## 13. 设计结论

本轮正确方向不是继续加功能，也不是迁移 `frontUI` 原型代码，而是：

> **在现有桌面前端功能闭环上，以 `frontUI` 的骨架与气质为参考，把 renderer 打磨成更像 Windows 本地聊天客户端的产品界面。**

这条路线满足三件事：
- 保留“桌面聊天工具”直觉
- 保持“分析 / 推演能力”可见但克制
- 不破坏当前已经跑通的产品主链路
