# “如果那时” Windows 桌面应用设计

日期：2026-04-06

## 1. 目标

将当前纯后端 MVP 演进为一个可发布的 Windows 桌面应用。

最终用户体验目标：

- 用户下载 release 后，双击 `.exe` 即可打开应用窗口
- 用户无需手动启动 API、worker 或命令行
- 应用内可完成：
  - 配置模型
  - 导入聊天记录
  - 后台分析
  - 浏览历史聊天
  - 选择历史消息改写并发起推演
  - 在聊天视图中查看反事实分支结果

模型策略：

- 联网 API 为主
- 本地模型能力需要预留产品与架构入口，但不是首个交付版本的主实现目标

## 2. 已确认决策

### 2.1 技术路线

- 桌面宿主：Electron
- 前端界面：React + TypeScript
- 后端核心：保留现有 Python + FastAPI + SQLite + worker
- 前后端连接策略：
  - 第一阶段前端通过本地 HTTP 调用 Python API
  - 但所有请求必须经由前端 service 层统一封装
  - 页面组件不得直接耦合本地 API 地址

### 2.2 产品形态

- 单窗口桌面应用，不做多窗口
- 主界面为三栏结构
- 设置与导入使用弹窗或抽屉，不切整页
- 首次启动自动弹出欢迎引导，但不强制阻塞
- 推演结果在右侧聊天区中以“分支视图”呈现，不跳到独立页面

### 2.3 首版范围

首版桌面应用采用中等完整版范围：

- 会话列表
- 导入聊天记录
- 聊天浏览
- 搜索/时间定位
- 分析进度展示
- topic / persona / snapshot 查看
- 选择历史消息并改写
- 单轮/短链推演
- 设置页（模型配置）

## 3. 当前系统与演进原则

当前项目已经具备：

- QQ 文本导入
- 解析与规范化
- 切段与 `merged_isolated`
- 段摘要、topic、persona、relationship snapshot
- cutoff-safe 检索
- `/simulations` 推演

桌面化不是推翻现有后端，而是将当前后端作为应用核心引擎嵌入桌面产品壳中。

演进原则：

- 最大复用现有 Python 核心逻辑
- 优先补齐后端产品化缺口，再推进前端桌面壳
- UI、桌面宿主、服务调用、后端进程管理严格分层
- 第一版优先做出完整闭环，再迭代桌面深度能力

## 4. 后端优先补齐的产品化缺口

在桌面前端开工前，先补齐以下四类缺口。

### 4.1 模型配置统一生效

问题：

- 当前 API / worker 的模型配置生效路径不统一
- 设置页保存后，运行中的 API / worker 不一定真实使用新配置

要求：

- 桌面应用中的模型配置必须成为单一可信来源
- 设置保存后，API 与 worker 必须以一致方式读取配置
- 配置变更后的行为必须可预测

设计要求：

- 配置模型时采用“产品字段”而不是暴露底层键值对
- 为未来“在线 API / 本地模型”双来源预留统一结构
- 若配置生效需要重启服务，则由应用负责触发和反馈

### 4.2 会话分析任务状态恢复

问题：

- 当前前端只在导入当次知道 `job_id`
- 应用重启后，不易恢复某个 conversation 当前或最近关联的 job

要求：

- 桌面应用重启后，仍可恢复分析中会话的状态展示
- conversation 与 job 的关联可查询
- 可恢复分析阶段、进度与完成状态

### 4.3 消息浏览与定位能力补强

问题：

- 当前消息查询接口足以支撑 MVP，但对桌面聊天式浏览仍偏基础

要求：

- 更适合聊天记录分页与定位
- 支持搜索、时间跳转、围绕消息定位上下文的能力扩展
- 第一版允许从基础能力起步，但接口需朝聊天式浏览演进

### 4.4 会话管理能力

问题：

- 当前缺少完整的删除会话、重跑分析等产品级操作

要求：

- 支持删除会话
- 支持重跑分析
- 为桌面版提供基本数据管理能力

## 5. 桌面应用总体架构

采用四层结构：

### 5.1 Electron Shell 层

职责：

- 创建与管理桌面窗口
- 管理应用生命周期
- 启动 / 关闭 Python API 与 worker
- 提供文件选择器、路径、版本、日志目录等桌面能力
- 承担后续打包、更新、诊断能力的宿主角色

约束：

- 不承载业务页面逻辑
- 保持为薄壳

### 5.2 Desktop Bridge 层

职责：

- 向前端暴露桌面专属能力
- 包括：
  - 选择聊天记录文件
  - 查询应用版本
  - 查询服务运行状态
  - 打开日志目录
  - 必要时触发后端重启

约束：

- 不承担主要业务数据查询
- 业务数据优先走 HTTP API

### 5.3 Frontend App 层

职责：

- 实现桌面应用产品界面
- 负责主布局、交互、弹窗、抽屉、视图切换、加载态与错误态

### 5.4 Frontend Service 层

职责：

- 统一封装所有后端接口调用
- 例如：
  - `getConversations`
  - `importConversation`
  - `getConversationMessages`
  - `getJobStatus`
  - `getTopics`
  - `getProfile`
  - `getTimelineState`
  - `getSettings`
  - `updateSettings`
  - `createSimulation`

约束：

- 页面组件不直接写原始网络请求
- 页面组件不感知本地服务地址

## 6. 进程模型

应用启动流程：

1. Electron 主进程启动
2. 创建主窗口
3. 自动拉起 Python API 进程
4. 自动拉起 Python worker 进程
5. 轮询本地 `/health`
6. 健康检查通过后，前端进入可用态

用户感知应为：

- 双击 exe
- 稍等片刻
- 直接进入应用窗口

而不是：

- 手动运行 Python
- 手动启动 API / worker

## 7. 信息架构与页面结构

### 7.1 主窗口结构

主窗口固定为三栏：

#### 左栏：主导航栏

首版保留：

- 聊天
- 联系人/会话
- 设置

保留扩展位：

- 本地模型
- 设备
- 实验入口

#### 中栏：会话列表栏

作用：

- 显示已导入联系人 / 会话
- 显示最近消息、最近时间、分析状态
- 作为整款产品的数据入口

空状态：

- 尚未导入聊天记录
- 提供导入与配置模型入口

导入后：

- 立刻出现一个联系人/会话项
- 即使分析尚未完成，也应显示“分析中”状态

#### 右栏：聊天主视图

包含三种模式：

- 未选中会话：欢迎占位态
- 正常历史聊天视图：浏览真实聊天记录
- 分支视图：展示反事实推演结果

### 7.2 首次启动体验

首次启动若检测到以下任一条件未满足，则自动弹出欢迎引导：

- 没有可用模型配置
- 没有任何 conversation

欢迎引导包含两个主动作：

- 配置模型
- 导入聊天记录

引导允许关闭，关闭后用户可从设置入口继续操作。

### 7.3 设置与导入交互

设置采用右侧抽屉。

内容包括：

- 在线 API 模型配置
- 本地模型预留配置
- 数据目录与运行状态
- 调试辅助信息

导入采用独立弹窗。

内容包括：

- 选择 QQ 导出文本
- 输入自己的昵称 / display name
- 发起导入

分析进度可以采用独立弹窗或局部面板呈现。

## 8. 关键交互流

### 8.1 首次启动

- 启动应用
- 后端自动拉起
- 进入主界面
- 弹出欢迎引导

### 8.2 配置模型

入口：

- 欢迎引导
- 左下角设置按钮
- 空状态快捷入口

要求：

- 保存后真实生效
- 可选做连接验证

### 8.3 导入聊天记录

入口：

- 欢迎引导
- 设置抽屉中的导入入口
- 会话空状态页按钮
- 会话列表顶部按钮

导入后：

- 会话立即加入列表
- 后台开始分析
- 会话显示分析进度

### 8.4 分析中体验

允许：

- 查看原始聊天记录
- 进行基础搜索

分析完成后解锁：

- 改写并推演
- topic / persona / snapshot 查看

### 8.5 浏览聊天

要求：

- 接近桌面聊天软件体验
- 气泡清晰
- 时间分组清晰
- 滚动顺畅
- hover 操作自然

### 8.6 选择消息并改写

触发方式：

- 鼠标 hover 在“我发出的历史消息”上
- 或右键消息

操作入口：

- `改写并推演`

打开改写面板后展示：

- 原消息
- 发送时间
- 改写输入框
- 推演模式选择
- 开始推演按钮

### 8.7 展示推演结果

结果在右侧聊天区进入分支视图。

分支视图包含：

- 顶部“当前正在查看分支”提示
- 原消息与改写内容说明
- 从分叉点开始的推演回复与后续短链
- 返回原始历史按钮

目标体验：

- 像在原聊天时间线上打开一条新分支
- 而不是跳到独立分析报告页

### 8.8 查看分析信息

通过顶部按钮打开右侧信息面板。

面板可切换：

- Topic
- Persona
- Snapshot / 关系状态

其定位是辅助理解，而非主交互主角。

## 9. 前端状态与数据流

### 9.1 状态分层

分为四类：

- 应用级状态
- 会话级状态
- 聊天视图级状态
- 弹层/抽屉状态

### 9.2 应用级状态

包括：

- API 是否健康
- worker 是否在线
- 是否首次启动
- 是否存在模型配置
- 是否存在 conversation

### 9.3 会话级状态

包括：

- 会话列表
- 当前选中会话
- 每个会话的分析状态
- 当前会话的 job 状态

### 9.4 聊天视图级状态

包括：

- 当前消息列表
- 搜索词
- 时间定位
- 当前 hover 消息
- 当前选中目标消息
- 当前视图模式：`history` / `branch`
- 当前 simulation 数据

### 9.5 弹层状态

包括：

- 欢迎引导
- 设置抽屉
- 导入弹窗
- 改写推演面板
- 分析信息面板

### 9.6 首启数据拉取顺序

前端启动后按顺序拉取：

1. `/health`
2. `/settings`
3. `/conversations`

据此判断是否需要自动弹出欢迎引导。

### 9.7 会话列表更新策略

- 初始化时拉取 conversations
- 导入成功后立即插入返回的 conversation
- 若返回 `job_id`，立即开始轮询 job
- 分析完成后刷新会话状态

### 9.8 分析进度轮询

对 `queued / running` job 执行轮询。

轮询结束条件：

- `completed`
- `failed`

展示位置：

- 会话卡片
- 当前聊天顶部状态提示

### 9.9 历史视图与分支视图切换

右侧聊天主区定义两种模式：

- `history`
- `branch`

切换规则：

- 在历史消息上发起推演后进入 `branch`
- 点击“返回原始历史”回到 `history`

## 10. 现有 Figma 导出前端的复用策略

`D:\frontUI\src` 当前更像静态原型。

可复用部分：

- 三栏骨架
- 侧边栏视觉结构
- 会话列表视觉结构
- 聊天气泡视觉结构
- 组件库基础设施

需要重构部分：

- `mockData`
- 假发送逻辑
- 仅适用于即时聊天器的数据结构假设
- 页面状态与 API 连接方式

结论：

- 视觉与布局可以复用
- 交互语义必须重做以贴合“历史聊天浏览 + 反事实推演”产品

## 11. 后端 API 适配结论

当前已有 API 已足够支撑第一版桌面产品主链路：

- 健康检查
- 会话列表
- 导入
- job 查询
- 消息查询
- topic / persona / timeline-state
- settings
- simulations

但为达到桌面产品要求，需要先补齐第 4 节所列四类缺口。

## 12. 实施顺序与里程碑

### 阶段 0：后端产品化补口

先补：

- 模型配置统一生效
- 分析任务恢复
- 消息浏览/定位补强
- 删除会话 / 重跑分析

### 阶段 1：Electron 桌面壳

完成：

- Electron 工程骨架
- 主窗口
- 启动 Python API / worker
- 健康检查
- 基础 bridge

### 阶段 2：前端产品壳改造

完成：

- 将现有 Figma 导出前端改造为真实应用
- 去除 mockData
- 建立模块化页面与 service 层
- 接入真实后端

### 阶段 3：核心产品流闭环

完成：

- 首启欢迎引导
- 导入弹窗
- 分析中状态
- 历史聊天浏览
- 改写并推演
- 分支视图
- 分析信息面板

### 阶段 4：发布准备

完成：

- Windows 打包配置
- Python 运行时随应用分发策略
- 数据目录策略
- release 构建流程
- 基础日志与异常提示

### 里程碑

- `M1`：后端补口完成，具备桌面化基础
- `M2`：Electron 壳跑通，能自动启动后端
- `M3`：主产品流跑通
- `M4`：可打包发布

## 13. 非目标

以下内容不纳入本设计的首版实现目标：

- 多窗口工作流
- 首版即完成完整 IPC/BFF 重构
- 首版即完成本地模型完整运行支持
- 首版即完成自动更新、崩溃诊断、安装器高级体验
- 多分支树状并行对比界面

## 14. 设计结论

本设计确认的核心路线是：

- 先补齐后端产品化缺口
- 再做 Electron 桌面壳
- 再将现有 React UI 改造成真实产品前端
- 最终形成一个可打包成 Windows `.exe`、双击即用、以聊天历史浏览与反事实推演为核心体验的桌面应用

这是当前项目从“纯后端 MVP”走向“可交付产品”的最稳妥路径。
