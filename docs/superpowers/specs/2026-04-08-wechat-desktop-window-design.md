# 微信风格桌面窗口化改造设计说明

日期：2026-04-08  
状态：草案已写成，待用户审阅确认

---

## 1. 背景

当前 `desktop/` 已经完成了以下基础：

- Electron 宿主可正常启动
- React + TypeScript 前端已切到 frontUI 主壳
- 主聊天界面已具备三栏结构
- 会话列表、消息浏览、分析侧栏、改写推演、分支视图已基本接回
- Electron 可自动拉起项目根目录中的 Python API / worker

但当前视觉层仍然存在明显“网页装进桌面窗口”的感觉，主要问题有两类：

1. **窗口层仍是默认系统窗口**
   - 原生标题栏
   - 默认菜单栏
   - 默认边框与系统窗口观感
2. **前端布局层仍是居中白色内容卡片**
   - `FrontAppShell` 仍有 `max-width / max-height`
   - 整体不是全窗口铺满
   - 更像浏览器中的一个页面，而不是微信式桌面应用

用户本阶段的明确目标是：

> **把当前桌面端继续推进成“像微信一样的桌面应用窗口”，让用户打开后看到的是一个完整桌面程序，而不是带系统菜单栏和网页居中白卡片的开发态壳。**

---

## 2. 本阶段目标

本阶段的目标不是“把所有业务都做完”，而是完成**桌面窗口产品化**的关键一层。

成功标准如下：

1. 打开桌面应用后，不再看到默认 `File / Edit` 菜单栏
2. 不再使用系统原生标题栏作为主要视觉头部
3. 应用主界面改为**全窗口铺满**
4. 顶部使用我们自定义的微信风格标题栏
5. 支持桌面应用最基本的窗口操作：
   - 最小化
   - 最大化 / 还原
   - 关闭
   - 标题栏拖动窗口
6. 当前 frontUI 主聊天壳继续作为内容主体，不破坏既有真实业务能力

---

## 3. 非目标

本阶段**不包含**以下内容：

1. 输入框真实发送能力
2. `contacts / files` tab 的真实业务接入
3. 所有弹层（设置、导入、欢迎引导）完全重做成微信视觉
4. 所有旧组件一次性删除
5. 所有分析/改写/分支面板在本阶段彻底 frontUI 化

换句话说，本阶段聚焦于：

> **先把窗口层做对，让应用整体看起来像真正桌面软件。**

---

## 4. 方案对比

### 方案 A：只改前端布局，不改 Electron 窗口

做法：

- 保留原生标题栏和菜单栏
- 只把 `FrontAppShell` 从居中卡片改成全窗口铺满

优点：

- 改动小
- 风险低
- 交付快

缺点：

- 仍然不像微信桌面应用
- 系统菜单栏和原生窗口边框仍然很出戏

### 方案 B：无边框窗口 + 自定义标题栏 + 全窗口铺满

做法：

- 去掉默认菜单栏
- Electron 改成无边框窗口
- React 自绘顶部标题栏
- 用 IPC 提供最小化 / 最大化 / 关闭 / 状态同步
- 主内容区全窗口铺满

优点：

- 最接近微信桌面版体验
- 窗口层与内容层统一
- 为未来 release 产品化打下正确基础

缺点：

- 要补一层 Electron IPC / preload 能力
- 要处理标题栏拖拽区与按钮点击区

### 方案 C：一步做到完整微信桌面拟态

做法：

- 在方案 B 基础上再全面重做弹层、控件、面板、交互细节

优点：

- 最完整

缺点：

- 范围太大
- 容易拖慢当前主线

### 最终选择

本阶段采用 **方案 B**：

> **无边框窗口 + 自定义标题栏 + 全窗口铺满 + 保留现有功能接线。**

---

## 5. 总体架构

本阶段仍然维持当前三层架构：

### 5.1 Electron 宿主层

负责：

- 创建桌面窗口
- 隐藏系统菜单栏
- 管理窗口状态
- 暴露窗口控制能力给前端

主要文件：

- `desktop/electron/main.ts`
- `desktop/electron/ipc.ts`
- `desktop/electron/preload.cts`

### 5.2 React 渲染层

负责：

- 自定义标题栏
- 主界面布局
- 窗口状态 UI 显示
- 继续承接 frontUI 主聊天界面

主要文件：

- `desktop/src/App.tsx`
- `desktop/src/frontui/AppShell.tsx`
- `desktop/src/frontui/*`

### 5.3 Python 后端层

职责不变：

- API
- worker
- 导入 / 分析 / 推演 / 读取数据

本阶段不改后端协议。

## 6. 详细设计

## 6.1 Electron 窗口层改造

目标：

- 去掉默认菜单栏
- 使用无边框窗口
- 允许 React 完整控制顶部标题栏视觉

建议改造点：

### `desktop/electron/main.ts`

当前：

- 使用标准 `BrowserWindow`
- 有默认标题栏
- 未显式处理菜单栏

目标改为：

- 关闭默认菜单栏
- 使用无边框窗口配置
- 保留窗口最小尺寸约束
- 保留现有 preload / contextIsolation 方案

窗口改造后仍由 Electron 主进程负责：

- 启动本地后端
- 加载 renderer
- 关闭前终止 API / worker

---

## 6.2 窗口控制 IPC

前端需要触发以下桌面行为：

- 最小化窗口
- 最大化窗口
- 还原窗口
- 关闭窗口
- 读取当前是否已最大化

因此需要新增一组 IPC 通道，由 preload 向前端暴露。

建议能力：

- `window.minimize()`
- `window.toggleMaximize()`
- `window.close()`
- `window.getState()` 或 `window.isMaximized()`

如果后续要做更精细体验，可继续补：

- `window.onStateChange(...)`

但本阶段不是必需项，初版允许以前端重新拉取状态为主。

---

## 6.3 自定义标题栏

建议新增组件：

- `desktop/src/frontui/WindowTitleBar.tsx`

职责：

- 占据窗口最顶部
- 提供拖拽区域
- 显示应用名或当前页面标识
- 放置窗口控制按钮

建议视觉方向：

- 与微信桌面版一致，保持简洁、低干扰
- 颜色与主界面统一，不额外做明显分层
- 按钮 hover/active 状态可参考桌面应用常见规范

标题栏区域应分成两种区域：

1. **可拖拽区**
   - 标题空白区域
   - 应用名区域
2. **不可拖拽区**
   - 窗口按钮
   - 搜索/操作按钮
   - 任何可点击控件

---

## 6.4 主壳布局改造

当前 `FrontAppShell` 问题：

- 外层居中
- 有 `max-w-[1200px]`
- 有 `max-h-[800px]`
- 看起来像一个网页内容卡片

目标改造：

- 根容器铺满整个 Electron 窗口
- 结构变为：

```text
WindowRoot
├─ WindowTitleBar
└─ MainContent
   ├─ Sidebar
   ├─ ChatList
   ├─ ChatWindow / BranchView
   └─ AnalysisInspector（按需）
```

核心原则：

- 不再居中一个白色卡片
- 不再人为限制主界面最大宽高
- 左中右三栏直接填满窗口

这样才能获得更像微信桌面应用的结果。

---

## 6.5 样式策略

本阶段样式处理原则：

1. frontUI 主壳仍然是主视觉母版
2. 旧分析/改写/分支面板允许暂时沿用已有样式
3. 自定义标题栏样式独立实现，不污染业务组件
4. 最大化与还原两种状态下，标题栏和内容区都不应出现错位

建议新增的样式范围：

- 标题栏
- 窗口控制按钮
- 无边框窗口下的顶部/边缘处理
- 全窗口铺满后的主壳 spacing 调整

---

## 7. 数据与交互流

本阶段不会新增业务数据协议，但会新增窗口控制交互流。

### 7.1 窗口控制流

1. 用户点击标题栏上的最小化 / 最大化 / 关闭按钮
2. React 调用 preload 暴露的方法
3. Electron 主进程执行窗口控制
4. 若需要，前端刷新窗口状态

### 7.2 标题栏拖拽流

1. 用户在标题栏拖拽区按下并拖动
2. Electron 将该区域识别为可拖拽区域
3. 窗口移动

### 7.3 业务功能流

原有业务流保持不变：

- 启动 -> 后端健康检查
- 设置读写
- 导入聊天记录
- 会话列表与消息列表读取
- topics/profile/snapshot
- 改写并推演 / 分支视图

## 9. 风险与缓解

### 风险 1：无边框窗口后拖拽与点击冲突

缓解：

- 只让标题空白区 draggable
- 所有按钮明确标记为 no-drag

### 风险 2：最大化状态下布局异常

缓解：

- 明确增加 maximized 状态样式分支
- 前端标题栏与主壳同时适配

### 风险 3：现有旧面板与新窗口风格暂时不完全统一

缓解：

- 本阶段接受 Mixed 状态
- 后续单独做分析/改写/分支视觉统一

### 风险 4：窗口层改造影响当前 Electron 启动稳定性

缓解：

- 不改后端拉起逻辑
- 不改 API/worker 路径解析逻辑
- 主进程测试与手工验证一起做

## 10. 分阶段实施建议

### Phase 1：窗口壳改造

- 去菜单栏
- 改无边框窗口
- 补窗口控制 IPC / preload

### Phase 2：标题栏落地

- 新增 `WindowTitleBar`
- 接入窗口按钮与拖拽区

### Phase 3：主壳全屏化

- `FrontAppShell` 去掉居中卡片
- 改为全窗口铺满

### Phase 4：收尾验证

- 测试
- GUI 手工验证
- 记录仍未 frontUI 化的面板区域

---

## 11. 最终结论

本阶段采用的正式方案为：

> **在不改变后端协议和核心业务流的前提下，把当前 Electron + frontUI 主界面推进到“微信式桌面窗口”形态：去默认菜单栏、改无边框窗口、引入自定义标题栏，并让整个主界面全窗口铺满。**

这一步完成后，项目的桌面化质感会从：

> “桌面窗口里打开了一个网页”

提升为：

> “这本身就是一个完整的桌面应用”。
