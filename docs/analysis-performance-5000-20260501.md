# 5000 条消息分析性能复测报告

日期：2026-05-01

本报告使用 MVP 测试集 `D:\聊天记录-mvp测试集\聊天记录-5000条.txt` 复测当前分析链路，并与旧链路报告 `docs/analysis-performance-5000-20260426.md` 对比。复测运行在隔离数据目录中，没有修改应用正常使用的 `.data` 数据库。

## 结论

速度有明显提升。旧链路 worker 墙钟耗时 `1204.812 s`，当前链路耗时 `572.206 s`，减少 `632.606 s`，总等待时间下降 `52.51%`，约为旧链路的 `2.11x` 速度。

精度没有发现结构性下降证据，但不能做“严格证明”。本次旧链路的原始 `result.json` 和数据库快照不在当前工作区，只能与旧报告中的汇总指标对比，不能逐条做语义 diff。当前链路完成了 5,000 条消息、47 个分段、47 个分段摘要、2 个人设画像、47 个关系快照，且没有空摘要/空画像/空快照。Topic 数从旧链路 13 个变为 10 个，topic link 从 61 个变为 51 个；当前输出看起来是更集中地合并话题，但没有旧输出内容和人工标注，不能判定这是精度下降。

## 阶段耗时

当前链路中 `snapshots` 分支会与 `topic_resolution` / `persona` 分支重叠执行，所以新链路阶段耗时不能再简单相加为总墙钟耗时。

| 阶段 | 旧链路秒数 | 当前链路秒数 | 变化 |
| --- | ---: | ---: | ---: |
| summarizing | 374.135 | 121.832 | -67.44% |
| topic_resolution | 446.489 | 389.729 | -12.71% |
| topic_merge_review | 24.061 | 23.849 | -0.88% |
| persona | 29.561 | 36.255 | +22.64% |
| snapshots | 330.114 | 350.664 | +6.23% |
| parsing | 0.427 | 0.506 | +18.50% |
| segmenting | 0.009 | 0.010 | +11.11% |

最明显收益来自 `summarizing`：47 个分段摘要从串行调用改为受并发上限控制的窗口式并发后，阶段耗时从 `374.135 s` 降到 `121.832 s`，约 `3.07x`。

`snapshots` 自身没有变快，但它现在与 topic/persona 分支重叠，因此不再完整叠加到用户等待时间上。

## 进度显示

worker 日志中的 `overall` 进度在 parsing 后仍会很快到 `97%`，这一点与旧报告指出的问题类似；但当前 payload 中已经包含结构化 `stages`，各阶段都能独立显示 `completed_units / total_units`。因此真实 UI 应依赖 `stages` 展示阶段级进度，而不是只看单一 overall 百分比。

## 判断

本次 TODO 第一条优化在同一 5000 条测试集上有明确速率收益：总等待时间下降超过一半。基于当前可验证指标，没有发现精度结构性下降：核心分析产物数量完整，输出非空，confidence 水平正常。

但“精度没有下降”还不是严格回归结论。要严格证明，需要保留旧链路完整输出，或建立人工标注/抽样评分集，再对 segment summary、topic 覆盖、persona 与 snapshot 做逐项对比。
