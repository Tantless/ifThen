# 后端接口与前端使用盘点

> 审计时间：2026-04-10  
> 审计范围：`src/if_then_mvp/api.py`、`desktop/src/**`、`desktop/electron/**`

## 结论速览

- 后端共暴露 **18 个 API 路由**（见 `src/if_then_mvp/api.py:77-490`）。
- React renderer 真实接入了其中 **11 个接口**。
- Electron 桌面壳额外使用了 **1 个健康检查接口**：`GET /health`（见 `desktop/electron/main.ts:3-19`）。
- 当前**未被前端/桌面壳接入**的后端接口有 **6 个**。
- 当前前端里明确仍是 **mock / 占位** 的功能模块有 **3 类**：
  1. 通讯录 tab
  2. 文件 tab
  3. 聊天窗口“发送消息”本地追加
- 另外还有 **1 类“由后端返回，但以前端 mock 气泡渲染”的功能**：改写推演结果消息。它**不是纯前端伪造数据**，但 UI 层明确把它标成了 `source: 'mock'`。

| 聊天输入框“发送消息” | **本地 mock 追加** | 发送动作只在前端构造 `mock-local-*` 消息并塞进 `mockMessagesByConversation`，没有任何后端发送接口调用 | `desktop/src/frontui/ChatWindow.tsx:286-291`；`desktop/src/App.tsx:1173-1195` |

## 4.2 不是纯 mock，但在 UI 中按 mock 消息渲染的功能

| 前端功能 | 实际来源 | 应如何理解 | 证据 |
|---|---|---|---|
| 改写推演后的分支消息 | **后端 `/simulations` 返回** | 数据来源是真实后端推演，不是前端胡编；但它不是原始聊天记录，也不会写回消息表，所以 UI 明确标成 `source: 'mock'` | `desktop/src/App.tsx:326-342,1078-1084`；`desktop/src/lib/frontUiAdapters.ts:127-166` |

> 也就是说：**“改写推演结果气泡”不是纯前端 mock 功能，而是“后端驱动的非持久化模拟结果”**。如果只统计“前端自己伪造”的功能，不应把它算进纯 mock 模块里。

## 六、建议你接下来最值得补齐的前后端断层

按“后端已做完、前端还没接”的角度，优先级建议如下：

1. **重新分析入口**  
   - 后端已有：`POST /conversations/{conversation_id}/rerun-analysis`
   - 前端现状：已有 service，但没有 UI 入口

2. **会话删除入口**  
   - 后端已有：`DELETE /conversations/{conversation_id}`
   - 前端现状：完全未接

3. **消息上下文查看**  
   - 后端已有：`GET /messages/{message_id}/context`
   - 前端现状：改写场景里很适合接一个“查看上下文”面板

4. **segments 可视化 / 分段视图**  
   - 后端已有：`GET /conversations/{conversation_id}/segments`
   - 前端现状：尚无展示模块
